Architecture Review · No Code · Ground Truth: 2026-07-13 Audit

Institutional discipline, retail-appropriate infrastructure. Not the reverse.

Every module below exists because a specific, cited audit finding needs a structural home, not because a bigger platform needs more parts. The target is one repository, one shared library, one config system, one test suite — sized for a solo operator running ₹4L paper capital toward a ₹1.6–10L live book, built so that live trading is a broker-adapter swap on top of paper, not a second system.

Architecture Blueprint

Clean/Hexagonal architecture: a pure, deterministic domain core (signal math, sizing math, risk math — no I/O, fully unit-testable) surrounded by ports (interfaces the core depends on: DataProvider, BrokerAdapter, Clock, Repository, AlertSink) implemented by adapters (yfinance client, NSE scraper, Zerodha/Angel One clients, SQLite repository, Telegram bot). Dependencies point inward only — adapters depend on ports, domain code never imports a concrete adapter. This single rule is what makes paper trading and live trading the same code path with a different adapter plugged in, and what makes the domain layer testable without a network connection.

Why this shape, specifically

The audit's most consequential findings — fabricated universe (F9), unresolved survivorship bias (F1), cost-model drift across strategies (F4), duplicated params (F8) — are all instances of the same root cause: no boundary existed between "logic" and "data/broker/config access." A strategy script could reach out and hardcode its own universe, its own cost constant, its own params, because nothing stopped it. A hexagonal boundary makes that structurally impossible going forward: a strategy in the domain core can only ask a DataProvider port for a universe — it cannot define one itself.

Scale calibration — a deliberate decision, not an oversight

"Institutional-grade" describes engineering discipline (correctness, reproducibility, auditability), not infrastructure scale. A weekly-rebalance, four-strategy, ₹4L-book system does not need Kubernetes, a distributed data warehouse, or a multi-region deployment — it needs Docker Compose on one small VPS, SQLite promoted to Postgres only if concurrent writers ever appear, and a single-node scheduler with a heartbeat. Building institutional-scale infrastructure for retail-scale capital would itself violate the brief's own rule: never add complexity without justification. Every infra choice below states its upgrade trigger explicitly, so scale-up is a documented decision, not a rewrite.

Target Architecture

One repository. One package tree. Deployable services are thin processes that import the shared library and wire concrete adapters via dependency injection at startup — they contain no business logic themselves.

Dependency direction (no cycles, enforced by CI import-linter)

services → quantos_core.* → quantos_core.utils. Within quantos_core: risk/portfolio/execution depend on strategies/factors/data ports, never the reverse. brokers depends on nothing else in the tree except utils. experiments and tools are leaves — nothing depends on them, and they may import quantos_core but never the other way. This asymmetry is what stops a research script from silently becoming a production dependency, which is exactly what happened when 18 audit/research markdown docs and the production strategy code ended up in the same flat folder.

Migration Strategy

Strangler-fig, not rewrite. The existing, working, research-validated strategy logic in momentum_backtest.py/paper_trader.py is correct enough to trust (proven look-ahead-free, proven deterministic) — the problem is everything around it, not the signal math itself. Migration preserves that logic byte-for-byte while relocating it behind new interfaces, verified at every step by regression tests asserting old-output equals new-output before any behavior is allowed to change.

  1. Git first, before anything else moves. git init at the new quantos/ root, commit the current AlgoTrader + trading_backtests trees verbatim as the baseline tag. Every subsequent step is a reviewable diff against that baseline — the single highest-leverage, lowest-risk first move available, and the audit's most basic unmet precondition.
  2. Characterize before refactor. Write regression tests that run the CURRENT scripts against fixed cached inputs and pin their exact output (equity curve, trade log) as golden files. These tests exist to fail loudly if a "pure refactor" accidentally changes behavior — they are deleted or superseded once the real test suite (§7) covers the same ground.
  3. Extract, don't rewrite, one module at a time in the dependency order of §12 — config and storage first (nothing depends on business logic), data platform second (strategies depend on it), strategies third (mechanical port behind the new interface, logic unchanged), then portfolio/risk/execution/brokers.
  4. Each extraction ends with the golden-file regression test still passing. If a fix is warranted mid-extraction (e.g. F9's fabricated universe), it happens as a separate, explicitly-labelled commit that intentionally breaks the golden file, with the new expected output re-pinned and the reason recorded in an ADR — never silently.
  5. Old entry points stay callable until their replacement service has run in parallel for at least one full rebalance cycle with matching output, then are deleted. No flag day.

06 / 15

Engineering Standards

SOLID, concretely (not as a slogan)

  • Single Responsibility. A strategy computes signals. It does not fetch its own data, does not size its own positions, does not know what a broker is. Today's momentum_backtest.py does all four in one 615-line file — the split is the whole point of §3's layering.
  • Open/Closed. New strategy = new class implementing the Strategy protocol + one YAML entry. Zero existing files change. Today, adding strategy 17 means copy-pasting a whole script.
  • Liskov. Any BrokerAdapter (Paper, Zerodha, Angel One) must be substitutable in the execution engine with no caller-side branching. This is the mechanism that makes paper trading a real rehearsal of live.
  • Interface Segregation. BrokerAdapter is not one fat interface — order placement, market data, and account/margin queries are separate protocols a given adapter composes, so a data-only context never has to stub out order methods it can't support.
  • Dependency Inversion. Domain code depends on DataProvider/BrokerAdapter abstractions it owns; adapters live outside and implement them. This is the enforced rule in §1, not aspirational.

Functional core, imperative shell

Everything in factors, strategies (signal logic), portfolio (sizing math), and risk (limit math) is pure: same inputs → same outputs, no hidden state, no I/O. This is what makes the --selftest pattern the audit found and approved of (assert-based, hash-seed-varying determinism checks) generalizable into real property tests instead of five hand-written scripts. All I/O — network calls, file writes, broker orders — lives in adapters at the shell, called only from services.

08 / 15

Infrastructure Standards

Git & branching

Trunk-based. Protected main. Short-lived feature branches, one concern each. Solo-operator "review" = a mandatory cooling-off checklist (does this touch the frozen-strategy list per the prospective-validation rule? does it change a golden-file output? is there a test?) enforced by a PR template, not a second human.

CI/CD pipeline

Push → lint + typecheck + test (§7, PR-blocking subset) → build Docker image, tag with commit SHA → (main only) manual approval → deploy to the single host by pulling the tagged image. Rollback = redeploy the previous SHA-tagged image; image immutability makes this a one-command action, not a re-debug session.

Containers — right-sized

Docker Compose, one host (a small VPS or the existing laptop running headless), one container per service (data_service, paper_trading_service, risk_monitor_service, later live_trading_service), one Postgres/SQLite container for storage. Explicit non-goal: Kubernetes. Upgrade trigger, stated in advance so it's a decision not a drift: multiple hosts needed for latency/redundancy reasons, or more than ~5 services needing independent scaling — neither applies at 4-strategy, weekly-rebalance scale.

Environments

dev (local, fixtures/cassettes, no network) · paper (real data providers, PaperBrokerAdapter only) · live (real data providers, real BrokerAdapter, gated by §9's compliance checklist). Config layering (§5, config module) selects environment; the same Docker image runs in all three — only config and the injected adapter change.

Secrets

Broker API keys and any future LLM keys live in the deploy host's OS-level secret store (Windows Credential Manager locally, or the VPS provider's secret manager if hosted) — never in .env committed to git, never in a config YAML. .env.example's current dead-code status (audit-confirmed: nothing loads it) is itself the warning sign this needs an actual loader, not just a template file. Rotation: broker keys rotated on every SEBI-mandated interval at minimum; a rotation runbook lives in docs/, not tribal knowledge.

Versioning & releases

SemVer tags on quantos_core. Strategy versions are separate from platform versions — a strategy's params/logic version lives in strategies_registry and is what the Prospective Validation freeze (CONTEXT.md's own rule) actually gates, so "restart the validation clock" has a precise, git-diffable trigger instead of a judgment call.

Risk Controls

Design decision: VaR/CVaR are computed and reported (§ analytics/monitoring) but are not the primary hard gate at this capital scale. VaR needs return-history depth and regime stability a <1-year live track record doesn't have yet — a false-precision statistical gate on immature data is worse than a simple, robust limit. The primary gates below are deliberately blunt.

Single-name position limit15% of book NAVPre-trade gate rejects the order; no partial override without explicit config change + ADR
Sector exposure limit30% of book NAVPre-trade gate rejects new same-sector orders past limit

Daily loss — soft−3% NAV intradayHalt new order generation, alert operator, existing positions untouched
Daily loss — hard−5% NAV intradayGlobal kill switch engages (see below)

Broker health-check3 consecutive failed heartbeats within 60sSAFE_MODE: block new orders, alert; existing positions deliberately left alone — auto-flattening into a disconnect-induced bad quote is its own failure mode
SEBI algo-order-rate / Algo-ID tagPer current registration status (§13's compliance gate)Pre-trade gate rejects any order missing a valid Algo-ID tag once registration is required

The kill switch

One global, persisted (survives process restart — a flag in storage, not memory) boolean, checked by the pre-trade risk gate before every single order, in every service, with no bypass path. Settable automatically (daily-loss hard breach, data-quality failure, broker health failure) or manually via a CLI the operator can run from anywhere. This is the single control the audit found most conspicuously absent (F6) and the one piece of infrastructure worth building before any other risk-engine sophistication.

Critical Path

Hard dependency order — each item genuinely blocks the next, independent of calendar time:

git baseline → config/storage skeleton → data platform (PIT fix) → strategy interface migration → risk engine → execution engine + paper broker → observability → SEBI compliance closed → live broker adapter → chaos-tested hardening

Two items are NOT on this critical path and can run in parallel at any point without blocking it: the SEBI compliance checklist (pure verification, no engineering dependency — should start immediately, in parallel with Phase 0) and the size-factor regression the research audit already proposed (pure research, doesn't gate any engineering phase). Definition of Done

Applies to every module, every phase — nothing is "complete" without all of these, matching the audit's own standard of "never mark anything complete without repository evidence."

  • Code lives in quantos_core or a service, imported through its public interface only — no reaching into another module's internals.
  • Unit/property tests exist and are wired into CI, not gated behind a manual flag.
  • Typed exceptions only; every catch either re-raises or logs+alerts, never both silent and swallowed.
  • Config values sourced from config, zero magic numbers in the diff.
  • An ADR exists for any decision that overrides a default in this document (e.g. changing a risk threshold).
  • Golden-file/regression evidence attached to the PR showing behavior is unchanged, or the change is explicit and justified.
  • Structured log lines and at least one metric emitted for anything that runs unattended.
  • Documented failure mode + recovery path for anything that touches a network call, a broker, or capital.

Final Institutional Readiness Assessment

QuantOS's research discipline on its flagship strategy is genuinely ahead of its software maturity — that asymmetry is the whole story. The architecture in this document doesn't add capability the project lacks conviction for; it builds the four things a real trading operating system cannot exist without and that the audit found completely or almost completely absent: a portfolio-level risk engine with a kill switch (0/5 today), an execution engine with a real order lifecycle (dict mutation today), a single shared library in place of two disconnected codebases, and a point-in-time data platform that makes fabricated/current-only universes structurally impossible rather than a documented caveat.

None of Phases 0–7 require SEBI clearance or touch live capital — they can proceed in full regardless of how long compliance verification takes, and doing so is the correct order: an institutional-grade paper-trading rehearsal, proven under chaos testing, is worth more than early live capital on an unproven pipeline. Phase 8 is the only phase gated on an external, non-engineering dependency (the compliance checklist), and it is sequenced last on purpose.

Readiness verdict: not institutional-grade today (DRS 53/100, per audit), but on a coherent, evidence-driven, non-speculative path to it — every module in this blueprint exists to close a specific, cited gap, not to resemble a bigger platform. Confidence in this architecture: high for Phases 0–7 (pure engineering, no external dependency, directly traceable to audit evidence); dependent on operator follow-through for Phase 8 (the SEBI checklist has been open, unstarted, three months past deadline, since before this blueprint existed).
