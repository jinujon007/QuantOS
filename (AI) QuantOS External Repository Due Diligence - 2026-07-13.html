Executive Summary

All six repos were read against one question: does this close a gap the 2026-07-13 audit already found with a citation, without adding infrastructure QuantOS's ₹4L-paper / ₹1.6-10L-live solo book will never use? The honest answer, repo by repo: none of the six survive as a runtime dependency of the trading path. One (empyrical/pyfolio, surfaced via Awesome Quant) is worth importing for standard performance metrics. Everything else that earned a keep is a specific pattern — point-in-time data modeling, purged cross-validation, a market-impact cost formula, a turnover-controlled rebalancing algorithm, a look-ahead-bias guard — worth reading once and reimplementing in QuantOS's own ~50-200 lines, never worth pulling in as a package.

The single most consequential finding across all six: Microsoft Qlib's point-in-time (PIT) fundamentals model and point-in-time universe membership (constituent add/remove date ranges) are a near-exact structural fix for the audit's two worst-scored findings — F9 (fabricated 95-ticker universe) and F1 (survivorship bias, current-only universe fetch). Neither requires importing Qlib; both are reimplementable as a small SQLite/CSV table. This slots directly into the already-designed Phase 2 (Data Platform) of the existing blueprint — it doesn't change the architecture, it de-risks it.

The second most consequential finding is a rejection, not an adoption: TradingAgents' authors explicitly disclaim run-to-run determinism ("two runs of the same ticker and date can differ... expected for a research tool") and were still patching look-ahead data leaks in point releases as recently as mid-2026. That is a direct, structural conflict with QuantOS's own audited requirement — three consecutive byte-identical production runs, proven — and with the active Prospective Validation freeze on the momentum strategy. No LLM-agent trading-decision component belongs anywhere near the strategy path at this project's current stage, full stop.

Scale calibration, stated plainly: the brief asks this committee to think like Jane Street, Citadel Securities, Two Sigma, RenTech, HRT, IMC, AQR. Those firms would not build what four of these six repos offer for a ₹4L book either — Qlib's RL execution engine exists to slice institutional order sizes that move markets; QuantOS's largest single order at ₹1.6L live capital across 10 positions is ~₹16,000, an amount NSE liquidity absorbs without measurable impact. Thinking like an institutional engineer here means recognizing which institutional tools solve a problem this project doesn't have — not importing them anyway.

Repository Due Diligence

1. Microsoft Qlib

LicenseMITStars / Forks46.2k / 7.4kLast pushApr 2026 (active)

An institutional AI-oriented quant research platform (Microsoft Research), not a trading system — data → model → strategy → backtest → analysis, config-driven via YAML, built on PyTorch/LightGBM/XGBoost/MLflow plus a custom binary columnar data store. Two things stand out as genuinely load-bearing: a point-in-time (PIT) fundamentals database (announcement-date-keyed value triplets, so a query "as of date X" only sees what was actually public then) and point-in-time universe membership (constituent pools defined by date ranges, not a current snapshot). Both are direct structural fixes for QuantOS's F1/F9 bugs. The Alpha158/Alpha360 feature handlers and expression-DSL factor definitions are a reusable ergonomic pattern (declare a factor as a formula string, let an engine compute it) — useful once QuantOS's factor count grows past a handful, not today. TopkDropoutStrategy is a genuinely simple, decouplable turnover-controlled rebalancing algorithm directly applicable to QuantOS's weekly top-N rebalance. The Exchange backtest engine's quadratic market-impact cost term is a real formula worth knowing, not worth using at this order size. Reject outright: the RL-based intraday order-execution framework (TWAP/PPO agents slicing large orders — solves a problem QuantOS doesn't have), the MLflow/TaskManager online-serving infrastructure (built for teams running many models in parallel, not a solo operator with 4 strategies), and the custom binary data format (parquet/CSV is strictly sufficient at weekly-rebalance, ~500-name scale).

Verdict: REFERENCE ONLY. Read for the PIT data model, the rolling-window walk-forward task-generator pattern, and the TopkDropout algorithm; reimplement each in native QuantOS code. Never import Qlib itself — its ML-infra weight is disproportionate by roughly two orders of magnitude to what a 2,100-line solo codebase needs.

2. TradingAgents

LicenseApache-2.0Stars / Forks~87-93k / ~17kLast releasev0.3.1, Jul 2026 (active)

A LangGraph-orchestrated multi-agent LLM framework simulating a trading firm: four analyst agents (fundamentals/sentiment/news/technical) feed a bull/bear researcher debate, a trader agent, a risk-management debate team, and a portfolio-manager veto — a dozen-plus chained LLM calls per ticker per decision, across 11+ supported providers. Three findings settle this one decisively. First, the authors themselves explicitly disclaim determinism — documented, not inferred: "two runs of the same ticker and date can differ... expected for a research tool." QuantOS's own audit proved three consecutive byte-identical production runs as a hard requirement; this is a direct, stated incompatibility, not a tuning problem. Second, look-ahead data leaks were still being found and patched in the two most recent point releases (Alpha Vantage future-dated reports leaking into historical runs, stale-OHLCV acceptance) — for a tool whose value proposition is trustworthy backtesting, that is immature evaluation discipline, confirmed as of mid-2026, not a dated criticism. Third, no cost economics are disclosed anywhere and no real-money live track record exists — the docs explicitly disclaim it as "not a strategy with a fixed, replicable return." The one legitimately extractable idea — a typed, date-anchored data-fetch guard pattern rejecting stale/future-dated records — has nothing to do with LLMs and belongs in the data module regardless of anyone's opinion on agentic trading.

Verdict: REFERENCE ONLY for the look-ahead-guard pattern; effectively REJECT for everything agentic. This also directly reaffirms the project's own PRD, which already deferred an LLM signal layer for lack of Indian-retail precedent — this due diligence adds a second, independent reason: it would break the audited reproducibility guarantee the frozen validation protocol depends on.

3. Machine Learning for Trading (Stefan Jansen)

Confidence flag: the research agent hit a session limit before completing this repo and did not reach its own stated next step (checking the stable second-edition branch). One finding was captured before the cutoff and is reported as-is, unverified beyond that single observation: the current default branch may be a rolling rewrite with only chapters 1–10 published as code and chapters 11–27 present as roadmap text only. If this repo is evaluated further, target the second-edition branch specifically — that is the complete, well-established version referenced by the published 2020 O'Reilly book, not necessarily whatever is on main in 2026.

Known facts independent of the interrupted research: this is a book's companion code repository (Stefan Jansen, Machine Learning for Algorithmic Trading, 2nd ed.), organized by chapter — data sourcing, alpha-factor research (zipline/Alphalens/TA-Lib), linear/boosting/tree models, unsupervised learning, NLP/sentiment, deep learning (RNN/CNN/autoencoders/GANs), reinforcement learning. It is pedagogical, notebook-heavy code, not a packaged library — that much is true of every "book repo" in this category and does not need independent re-verification. Its value was never going to be as a dependency; it's a curriculum. The relevant question is which research methodologies it demonstrates that close QuantOS's audited gaps: purged/embargoed k-fold cross-validation for financial time series (De Prado-style — directly answers the audit's "single train/test split only" weakness), triple-barrier labeling and meta-labeling (a research-track idea for future strategy work, not the frozen momentum strategy), and Fama-French-style multi-factor regression (a direct extension of the CAPM/size-factor falsification chain the project already ran by hand).

Verdict: REFERENCE ONLY (curriculum, not dependency) — but purged k-fold CV and the multi-factor regression pattern individually earn ADAPT status as research methodology to reimplement in the validation/analytics modules, independent of the branch-stability question above.

4. Computational Finance Course (Lech Grzelak)

LicenseBSD-3-ClauseStars / Forks636 / 180Last commitMar 2024 (frozen, finished course)

A 14-lecture university computational-finance course (companion to Oosterlee & Grzelak's textbook): Black-Scholes fundamentals → implied volatility → jump processes → Heston stochastic vol → Fourier/COS pricing → Monte Carlo → hedging/Greeks → Bates/exotics. Real, runnable Python per lecture, but flat teaching scripts with no error handling, no tests, no packaging — read the math, don't import the file. QuantOS runs one options strategy (weekly index premium selling, rules-based, no pricing math today), so relevance is narrow and specific rather than broad: Lecture 4's Newton-Raphson implied-vol solver and Lecture 11's closed-form Black-Scholes delta/vega are legitimate, self-contained reference formulas for a future minimal Greeks-exposure check. Everything past Lecture 6 — Heston calibration, jump-diffusion, Fourier pricing, Bates, forward-start exotics — is unambiguously quant-desk-scale content with no line of sight to a rules-based retail seller with zero live capital in the options book.

Verdict: REFERENCE ONLY. Tier: Institutional Future — legitimate future building blocks for a Greeks-aware risk check if and when the options strategy scales toward real capital, nothing to build now.

5. Financial Models & Numerical Methods (cantaro86)

LicenseAGPL-3.0 (copyleft)Last pushOct 2024 (~21mo stale)MaintainerSingle, inactive

~20 Jupyter notebooks plus a real (but tightly coupled) src/FMNM/ package: Black-Scholes numerics, Heston, Lévy processes, PDE/PIDE solvers, Kalman filtering, mean-variance optimization. The one plausibly relevant piece — a Black-Scholes closed-form price and implied-vol solver — is call-only and missing delta/gamma/theta (only vega is implemented), so even the narrow useful slice needs to be substantially rebuilt, not lifted. The AGPL-3.0 license is worth flagging on principle: irrelevant while nothing is imported, but a hard constraint (copyleft, viral on distribution) if anyone were ever tempted to copy code verbatim rather than reimplement from the math.

Verdict: REFERENCE ONLY. Read the Black-Scholes formulas as a correctness check when writing an options-Greeks module from scratch; do not import any code from this repo given the license and the incomplete Greeks it actually ships.

6. Awesome Quant

LicenseNone found (unlicensed curation)Stars / Forks27.7k / — Last commitJun 2026 (actively curated)

Confirmed: a categorized link index, 18 categories, no code of its own — the awesome-list format. Its value is entirely as a one-time sourcing tool, and it earned its keep on that basis: it surfaced jugaad-data (NSE/BSE/RBI data, a candidate second source to reduce QuantOS's confirmed single-point-of-failure dependency on yfinance), zipline-reloaded/backtrader/Backtesting.py (alternative backtest engines), PyPortfolioOpt/Riskfolio-Lib (portfolio optimization), and empyrical/pyfolio (standard performance-metrics libraries). Notably, it does not list OpenAlgo or any Indian broker-execution wrapper — the broker-abstraction gap (already flagged in QuantOS's own PRD as a spike to evaluate) stays unresolved by this source; that lead came from elsewhere and remains the right one to chase.

Verdict: REFERENCE ONLY — a directory, not a dependency. Pull the shortlist above, evaluate each on its own merits, close the tab. No reason to track it as an ongoing source.

Anti-Patterns — What QuantOS Must Never Copy

  • Framework-as-platform lock-in (Qlib). Adopting Qlib's YAML-task-config-driven workflow, custom binary data format, or MLflow-backed online-serving layer would make QuantOS's own domain logic a tenant inside someone else's platform — exactly what the brief's "never let an external framework own core domain logic" rule exists to prevent. Every extracted Qlib idea above is a reimplementation for this reason, not an import.
  • Non-deterministic decision-making in the trading path (TradingAgents). The authors' own documentation disclaims run-to-run reproducibility. QuantOS spent real, audited effort (regime-cache fix, hash-seed sort-before-iterate fix) proving three consecutive byte-identical runs. Introducing an LLM anywhere upstream of a trading decision reintroduces the exact class of bug that effort just closed.
  • Treating research/teaching code as production-shaped (ML for Trading, Computational Finance Course, Financial Models & Numerical Methods). All three are notebook-first, untested, unpackaged by design — that's correct for their purpose and disqualifying for direct reuse. The failure mode to avoid isn't reading them; it's copy-pasting a Jupyter cell into momentum_backtest.py and calling it done.
  • Manufacturing relevance for content that doesn't have any. Roughly 70% of the Computational Finance Course and Financial Models & Numerical Methods content (Heston calibration, Lévy processes, PIDE solvers, Bates/exotics) has zero honest connection to a rules-based weekly options seller with no live capital. The discipline here is naming that plainly rather than padding this report with borrowed academic prestige.
  • Copying AGPL-3.0 code without accounting for its copyleft terms (Financial Models & Numerical Methods). Irrelevant today since nothing is being imported from it — worth stating once, explicitly, so it isn't rediscovered the hard way if someone later lifts a formula verbatim instead of reimplementing from the math.

08 / 12

Engineering Savings

Almost nothing here is reused as code, so "lines of code saved" is the wrong metric — the real saving is bugs avoided that QuantOS's own audit proves are expensive to find later. F1 (survivorship bias) and F9 (fabricated universe) were found by a manual audit after the fact; Qlib's PIT pattern is exactly the structural fix that would have made both bugs impossible to write in the first place, not just easier to catch. The look-ahead bias fix on 2026-07-12 (same-day signal/fill) was found and fixed manually — TradingAgents' typed data-fetch guard pattern is the generalized version of the same defense, applicable before the next such bug rather than after.

Rough estimate: 2-4 weeks of solo research/debugging time plausibly avoided by adopting the PIT-data pattern and purged-CV methodology outright, versus rediscovering both the hard way — which is precisely how F1/F9 and the single-split walk-forward gap were found in this project already. No estimate is offered for "time saved by not building a risk engine" or similar, because none of the six repos actually close that gap — the saving there is zero, and claiming otherwise would misrepresent this review.

Migration Integration — Four Stages, Mapped Onto the Existing Blueprint

Not a new roadmap. The existing blueprint's Phases 0-9 already stand; this maps the brief's requested 4-stage framing onto those phases and slots in exactly where an external idea strengthens a phase already planned. No phase gains new scope from this review — several gain concrete implementation detail they didn't have before.

  1. Stage 1 — Immediately after the validation freeze lifts= existing Phases 0-2
    Foundation, git baseline, quantos-core skeleton, Data Platform. New detail added by this review: PIT fundamentals + PIT universe (Qlib pattern) as the concrete implementation of Phase 2's F1/F9 fix; typed data-fetch guard (TradingAgents pattern) added to the same phase; empyrical/pyfolio adopted in Phase 1 for standard metrics; jugaad-data evaluated as a second DataProvider adapter candidate.
  2. Stage 2 — Before paper/live capital expansion= existing Phases 3-6
    Strategy Platform, Risk Engine, Validation Hardening, Execution Engine + Paper Broker. New detail: TopkDropout-style turnover control (Qlib) folded into Phase 3's strategy migration; purged k-fold CV + Fama-French regression pattern (ML for Trading) and the rolling-window task-generator pattern (Qlib) folded into Phase 5's validation hardening — this is where the review adds the most real substance, because it's where the audit's own weakest-scored subsystem (validation, single-split only) lives.
  3. Stage 3 — Institutional automation= existing Phases 7-8
    Observability + Automation, Live Broker Integration. No external idea from this review touches these phases — they're pure execution of the already-designed monitoring/services/brokers modules, gated on the SEBI compliance checklist exactly as already sequenced.
  4. Stage 4 — Institutional future / world-class= existing Phase 9 + explicitly deferred ideas
    Chaos/security/dashboard hardening, plus every row in §5 marked "Institutional Future": minimal BS Greeks module (if the options book scales), expression-DSL factor ergonomics (if factor count grows), triple-barrier/meta-labeling research track (if new-strategy research resumes), offline LLM devil's-advocate pass (if ever revisited). None of these are committed work — they're named and gated so a future decision to build them is deliberate, not drift.

What Survives

If QuantOS were built today from scratch by a world-class quantitative engineering team, which components, ideas, and architectural patterns from these six repositories would survive into the final design?

Five things, all of them ideas, none of them code: (1) point-in-time data modeling — fundamentals keyed by announcement date, universe membership keyed by constituent date ranges — as the default assumption for any data layer, not an afterthought bolted on after a survivorship-bias bug is found; (2) purged/embargoed walk-forward validation as the default backtest discipline instead of a single train/test split; (3) a typed, fail-loud data-ingestion boundary that rejects stale or future-dated records by construction; (4) a hard, non-negotiable determinism requirement on anything upstream of a trading decision — which is also the single clearest reason to keep every LLM-agent pattern out of the strategy path, this decade, at this capital scale; (5) the discipline demonstrated across this review itself — that reading six repos built by serious engineers and research teams, and concluding "borrow the idea, write fifty lines, walk away" is itself the institutional-grade move, not the compromise. A team that actually builds like Jane Street or Two Sigma doesn't import Qlib for a ₹4L book; it reads Qlib's PIT model on a Tuesday afternoon and has its own version by Wednesday.

Nothing in this document changes the target architecture, the phase sequencing, or the frozen validation protocol. It adds implementation detail to Phases 2 and 5, adds two new low-risk dependencies (empyrical/pyfolio, and conditionally jugaad-data) to Phase 1, and names five ideas explicitly deferred to "Institutional Future" so they're a documented decision if revisited, not a scope-creep risk today.
