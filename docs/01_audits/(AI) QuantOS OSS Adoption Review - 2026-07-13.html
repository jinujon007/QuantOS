Current QuantOS — Ground Truth

Full detail already exists in (AI) QuantOS Independent Audit - 2026-07-13.html and (AI) QuantOS Target Architecture Blueprint - 2026-07-13.html, both dated today. This review does not re-derive that audit — it takes it as given and asks one incremental question the audit didn't: do these two specific open-source projects change any of it? Summary of the parts load-bearing for this review:

Architecture

Two disconnected Python codebases (AlgoTrader/, 1,647 LOC; trading_backtests/, 4,513 LOC), zero shared library, zero git repository anywhere in the tree, zero broker connectivity, ₹0 live capital. 20 strategies total (4 live-paper-traded, 16 research-only), zero shared base class. A Target Architecture Blueprint already exists — a full quantos_core package design (data/factors/strategies/portfolio/risk/execution/brokers/analytics/validation/monitoring/storage) plus a 10-phase (0–9) migration roadmap — designed but not yet built.

Research maturity

Genuinely rigorous on one strategy: an honest CAPM → two-factor → three-factor falsification chain that found its own edge case (full-sample alpha p=0.002, loses significance on the fair post-2021 window, p=0.23). Applied to 1 of 20 strategies. Survivorship bias (current-only universe, F1) and a fabricated 95-ticker universe feeding two of four live strategies (F9) are both confirmed present, not merely risked.

Engineering maturity

Weak by the audit's own scoring: architecture 15%, testing 15%, risk 10%, execution 0%, deployment 0%. Correctness discipline is the standout positive — look-ahead bias and a Python hash-seed determinism bug were found and fixed with proof (3 consecutive byte-identical production runs). No dependency manifest for the flagship strategy. No shared code between the two codebases.

Operational maturity

One Windows laptop, no cloud, no containers, no CI, no monitoring/alerting, no dashboard. Scheduler has confirmed reliability gaps (a missing log day, a truncated run, a late trigger). SEBI/Angel One compliance checklist: 0 of 7 items closed, already three months past the enforcement deadline.

The one rule this entire review is bound by: Momentum's alpha model, ranking logic, universe, and parameters are frozen under the project's own Prospective Validation protocol (CONTEXT.md:14-36, active since 2026-07-13) until 13 clean weekly rebalances are observed — currently 0 of 13. Touching that logic restarts the clock. Every recommendation in this document was checked against that constraint before being written down; see §11.

01 / 13

Executive Summary

Qlib and TradingAgents sit at opposite ends of the "should we adopt this" spectrum for a different reason than stars, license, or code quality — both are perfectly respectable projects at what they're actually for. The mismatch is structural: QuantOS's real, current constraints are a Windows-only single laptop, an NSE/India universe neither project has native data support for, and a validation discipline (byte-identical reproducibility, frozen-strategy gating) that one of the two candidates is architecturally incapable of satisfying. Neither gap is a maturity problem that a future version fixes — they're design-time decisions in the source projects that point away from this use case.

Qlib, in one line

A mature, well-designed research platform whose individual ideas (Alpha158/360 factor formulas, TopkDropoutStrategy's rank-hold-drop shape, YAML dependency-injection config) are worth studying and reimplementing natively, but whose runtime (binary data store, C-extension build, no NSE adapter, explicitly "recommended on Linux," a Docker escape hatch this project can't use) is a debt trade at this scale — and the project itself shows real cooling signals (last push 3 months stale, ~48% of commits from one contributor, Microsoft's newer energy visibly going into RD-Agent, an LLM-agent layer built on top of Qlib).

TradingAgents, in one line

A well-engineered, actively maintained (47 commits in the trailing 30 days, real CI, healthy bazaar-style contributor base) multi-agent LLM research framework whose central design — an LLM Portfolio Manager rendering the final trade call with zero code-enforced position/exposure math — is the opposite of what a risk engine should be, and whose core output is, by its own maintainers' admission in-code, not reproducible run-to-run on identical inputs. That single fact makes it structurally incompatible with a project that treats byte-identical backtest reproducibility as a hard-won, proven requirement (§8 of the audit). This isn't a maturity gap TradingAgents will close with more releases — non-determinism on reasoning-tier models is a stated, permanent property of how it's built.

What this review actually changes

Six small, deferred, non-frozen-strategy items (§6) — mostly ideas and patterns, not code imports. Nothing on the critical path to the 13-week validation gate, the SEBI checklist, or any Phase 0–4 item in the existing Target Architecture Blueprint moves faster because of this review. What it does do: it closes the open question of whether either framework was a shortcut past the two genuinely hard unsolved problems — point-in-time NSE universe/fundamentals data, and a real portfolio-level risk engine. Both answers are no.

02 / 13

Qlib Evaluation

Verified against the live repo, official docs (qlib.readthedocs.io), and open GitHub issues — not summarized from memory. Full source citations in the underlying research pass.

Architecture & modules

Four layers: Data Layer (storage/retrieval/expression engine), Learning Framework (supervised models + RL), Workflow Layer (YAML-driven orchestration via qrun), Application Layer (strategy, portfolio construction, order execution/simulation). Modules are nominally independent but most real usage assumes the workflow/config glue (init_instance_by_config) — this is the crux of the adoption question in §13.

Data layer

Proprietary .bin binary format (adjusted OHLCV + factor column), built from CSV via scripts/dump_bin.py. An expression engine ($close, Ref(), Mean(), cross-sectional $rank) computes features declaratively. A separate point-in-time (PIT) database exists specifically to prevent restatement leakage in fundamental data — but the docs self-admit "the calculation of PIT is not performed in the optimal way." Critically: no evidence Qlib solves survivorship bias / delisted-ticker universe reconstruction — its instrument-universe files (csi300.txt etc.) read as static membership lists, the same class of problem as QuantOS's own F1/F9, not a solved one. Shipped data collectors cover Yahoo (US/CN), crypto, CN/US/BR index, Baostock — zero NSE/BSE adapter. Building one is a real multi-day build, not a config flag.

Feature / factor library

Alpha158 (~158 hand-engineered technical features) and Alpha360 (~360 raw normalized OHLCV values over a 60-day window, built for sequence models) are the flagship asset here. Both are expression strings in qlib/contrib/data/handler.py — genuinely portable: the math can be read out and reimplemented in plain pandas with zero Qlib runtime dependency.

Model zoo

25+ models under qlib/contrib/model/: LightGBM, XGBoost, CatBoost, linear/ensemble methods, and a full PyTorch neural stack (LSTM/GRU/ALSTM/GATS/Transformer/TCN/TRA/AdaRNN and more), all behind a common fit/predict interface, instantiated from YAML. Tree models are thin, easily-standalone wrappers; the time-series/RL variants are tightly coupled to Qlib's Dataset/Sampler classes and much harder to extract.

Backtesting engine

Event-driven, bar-by-bar simulation (Account/Position state carried forward through an Executor/Exchange loop) with vectorized data structures underneath for speed — structurally look-ahead-safe by forward-only calendar stepping, the same philosophy QuantOS's own T+1 fill invariant already implements. Cost model: open_cost/close_cost/min_cost/slippage-band parameters — directly comparable to (and no more sophisticated than) QuantOS's own transaction_costs.py. Daily-bar is the default and only well-supported path; intraday exists via a separate, materially heavier "Nested Decision Execution" framework, and an open, unresolved issue (#1846) confirms the Account class currently hardcodes daily frequency regardless of config — the intraday path doesn't actually work as documented.

Portfolio / strategy layer

TopkDropoutStrategy — hold K top-ranked names, drop the D worst each rebalance, replace with D best candidates. This is, functionally, exactly QuantOS's own Momentum design (hold top-10, weekly rebalance). Finding this pattern independently converged on in Qlib is a validation of the existing design, not a reason to import Qlib's implementation of it. WeightStrategyBase is a clean subclass-one-method extension point worth referencing as a shape when QuantOS builds its own Strategy interface (already planned, Blueprint Phase 3).

Risk engine

Thin — backtest performance analytics, not a live risk system. risk_analysis produces Sharpe/Information Ratio/MaxDD with-and-without-cost. No evidence anywhere in docs, source tree naming, or issue search of real-time correlation matrices, factor-exposure decomposition, position/sector limits, or a kill switch. This does not close QuantOS's F6 (no portfolio-level risk engine) at all — Qlib doesn't have that subsystem either.

Experiment tracking & workflow

Recorder is a thin wrapper around MLflow (MLflowExpManager) — mlflow ui works directly on its tracking directory. qrun config.yaml drives the full data→train→backtest→eval pipeline via a {class, module_path, kwargs} instantiation triple — a clean, genuinely reusable configuration pattern independent of the rest of Qlib. Reproducibility caveat: persistence is pickle-based, and the docs explicitly warn this can break across differing environments — a real risk, not a guarantee, for a project that treats exact reproducibility as a design requirement.

Deployment / online serving

Explicitly research-grade by the project's own admission: "generating orders for the next trading day is not supported due to limitations of public data," and a source comment flags delay_prepare as not properly implemented. Online serving gets you to "today's model prediction," not an order — the signal-to-broker gap is entirely unaddressed, same gap QuantOS already has.

Dependencies, license, Windows fit

MIT license, fully permissive. Python 3.8–3.12. Build requires a Cython C extension (rolling.pyx) unless a prebuilt wheel matches exactly — and the official docs plainly state "recommended to use Qlib in Linux," with Docker as the stated Windows escape hatch (unusable given this project's no-Docker footprint). Two open, unresolved Windows-specific build issues were found (#1769 — MSVC linker failure building the Cython extension from source; #1927 — joblib/Windows data-loading incompatibility). pip install pyqlib from a prebuilt wheel is probably fine; anything requiring source/editable install — likely, if lifting individual pieces — is the exposed edge, on exactly this project's platform.

Project health

46,164★, 7,352 forks, 463 open issues, MIT — but last push 2026-04-22, roughly three months stale relative to today, and no release found after v0.9.7 (2025-08-15) in an otherwise semi-annual cadence. Bus-factor concentration is real: one contributor (you-n-g) accounts for ~48% of top-contributor commits, top 3 for ~70% — classic single-maintainer-plus-lab pattern, not a broad community. Microsoft's visible newer investment is RD-Agent, an LLM-agent framework for automated factor/model discovery built on top of Qlib — read as Qlib itself settling into "stable infrastructure, not much new energy" phase.

The load-bearing question, answered: can any Qlib subsystem be lifted in isolation, or does using it mean buying the whole framework? Factor formulas and a single tree model: yes, cleanly separable — copy the math. Strategy-shape and config-triple patterns: separable as a design reference, not as imported code. The data layer, backtest engine, Recorder/MLflow wrapper, and online-serving stack: no — all assume Qlib's calendar, instrument-universe files, binary storage layout, and Position/Order/Account object graph. There is no middle path to "just use the good 20%" for those four.

03 / 13

TradingAgents Evaluation

Verified against the live repo (via GitHub API and raw source), the arXiv paper (2412.20138, v7 full text), and open issue/PR history — not summarized from memory.

Architecture

Genuine LangGraph StateGraph — a real DAG with conditional edges, compiled once and invoked per ticker+date. Split cleanly across six components (GraphSetup, ConditionalLogic, Propagator, Reflector, SignalProcessor, Checkpointer). Optional SQLite checkpointing gives crash recovery, keyed by ticker+date+graph-shape.

Agent framework / decision process

Four analyst agents (market, sentiment, news, fundamentals) each with real tool-calling loops over live data sources → Bull/Bear Researcher debate, fixed at 2 turns by config, not by convergence → Research Manager (structured-output investment plan) → Trader (transaction proposal) → three fixed-persona Risk Debators (Aggressive/Conservative/Neutral, similarly fixed-round, not consensus-terminated) → Portfolio Manager, the final gate. Reading the Portfolio Manager's source directly: it builds a prompt from the debate history and returns whatever a structured-output LLM call produces. No position sizing, no VaR/volatility calculation, no exposure-limit check, no stop-loss logic anywhere in this path — the entire "risk management" layer is three LLM personas arguing, not code. A community PR (#634, open, unmerged) explicitly proposes adding deterministic risk guardrails — its existence as an unmerged proposal is itself confirmation that shipped main has none.

LLM orchestration

Genuinely provider-agnostic: native clients for Anthropic, Google Gemini, Azure OpenAI, AWS Bedrock, plus OpenAI-compatible routing for OpenAI, xAI, DeepSeek, Qwen, GLM, MiniMax, OpenRouter, Mistral, Kimi, Groq, NVIDIA NIM, and Ollama (local, key-optional) via a lazy-loading provider factory. Two-tier model selection (deep_think_llm for the two highest-stakes nodes, quick_think_llm for the rest) — a clean, genuinely reusable abstraction independent of the rest of the app.

Signal generation

Real integrations, not stubs: Yahoo Finance / Alpha Vantage (price, fundamentals), stockstats (deterministic technical indicators), Reddit and StockTwits (sentiment), FRED (macro), Polymarket (prediction markets). All degrade gracefully on failure rather than crash. Vendor selection per data category is explicit and respected, not silently substituted.

Memory

Not a vector store today. Current main uses an append-only markdown log with return-based LLM reflection after outcomes are known — a real architectural simplification from an earlier ChromaDB-embeddings version (visible in closed issue #111) that has since been fully removed. Five most-recent same-ticker entries plus three cross-ticker reflections are injected into the Portfolio Manager's prompt as context.

Determinism — the decisive finding

The project's own configuration comment states plainly: no setting makes LLM output bit-identical across runs, and reasoning-tier models (the deep_think tier used for the two most consequential nodes) largely ignore temperature entirely. No seed parameter exists anywhere in the LLM client layer. This is admitted, permanent, and structural — not a bug to be fixed in a future release. QuantOS's own validation discipline treats exact backtest reproducibility as proven and load-bearing (3 consecutive byte-identical production runs, §8/§9 of the audit). These two properties cannot coexist inside the same decision path.

Token cost & latency

16–22 LLM calls per ticker per decision cycle (per a maintainer's own root-cause writeup in issue #750), with a documented 0% prompt-cache hit rate — an acknowledged, still-open inefficiency costing an estimated 30–40% overhead versus a cache-friendly design. A real user (issue #111) reported ~13 minutes wall-clock for a single ticker/date run. Stages run sequentially by default; a parallelization attempt (PR #634, unmerged) claims a 3.5× speedup on just the analyst phase. Weekly cadence tolerates minutes-scale latency per name, but this system was designed as a single-ticker deep-dive tool — running it across a 459-name Nifty 500 universe every week, even just to screen candidates, is not a scaling exercise, it's a different problem the architecture wasn't built for.

Production suitability

No brokerage/execution integration exists anywhere in the tree — the paper's own authors list live deployment as future work. Output stops at a rated signal (Buy/Overweight/Hold/Underweight/Sell); no order routing, fill/slippage modeling, or live circuit-breaker exists.

Research validity of the original paper

Backtest window: 5 months (Jun–Nov 2024), 6 large-cap US tech tickers (AAPL, GOOGL, AMZN, NVDA, MSFT, META), reporting Sharpe ratios up to 8.21 against weak baselines (buy-and-hold, MACD, simple MA crossovers) — figures characteristic of short-window/small-N overfitting, not validated live-trading evidence, and the authors report no out-of-sample or live results. Look-ahead bias from tool-fetched news is genuinely, actively addressed (news-window filtering, unit-tested — tests/test_news_lookahead.py, CHANGELOG 0.3.0/0.3.1 fixes) — real engineering discipline. But this cannot touch the deeper, unfixable-by-prompt-engineering problem: an LLM's parametric training-data knowledge of how a stock actually performed historically can leak into its reasoning regardless of what tools it's given. This exact critique class is documented independently in the literature (arXiv 2601.13770, "Look-Ahead-Bench"; arXiv 2605.24564) as an open, unsolved problem for LLM-based backtesting generally.

India-market fit

Basic NSE/BSE ticker-suffix awareness exists (benchmark_map handles .NS→^NSEI). A community fork attempt (issue/PR #1002, 53 commits ahead of upstream) confirms this is not production-ready for Indian equities out of the box — it had to add India-specific validation, defaults, and fallback paths, and still explicitly excludes live broker execution.

Dependencies, license, project health

Apache-2.0 (permissive, express patent grant, no copyleft). Heavy install: langchain/langgraph family, multiple provider SDKs, backtrader, and an unreferenced redis>=6.2.0 dependency (declared in pyproject.toml, zero references anywhere in the source tree — a loose end, not a red flag). Genuinely healthy project: 92,655★, 47 commits in the trailing 30 days, real CI (pytest across Python 3.10–3.13, ruff, clean-install smoke test), 40+ external contributors credited in a single release's changelog — the opposite maintenance-risk profile from Qlib.

The load-bearing finding: project health, engineering discipline, and provider-agnostic LLM plumbing are all genuinely good here — better, in several respects, than Qlib's. None of that matters for this decision, because the one property that does matter — can the same inputs reproducibly produce the same trading decision — is answered no, by the project's own documentation, as a permanent architectural property rather than a fixable gap.

Recommended Components

Six items, all either applicable only to non-frozen strategies, or deferred to a not-yet-built module, or deferred to post-live-capital per the PRD's own existing non-goals. None require installing Qlib or TradingAgents as dependencies.

1 · Alpha158 factor formulas (selective)

Port the momentum/quality-relevant subset of Alpha158's expression-string formulas into native pandas functions, for use in Quality Factor (not Momentum, frozen). Real, immediate time saved — these are documented, tested formulas, not something to re-derive from scratch.

2 · TopkDropoutStrategy / WeightStrategyBase shape

Reference the rank → hold-K → drop-D → reweight decomposition when building the Strategy interface (Blueprint Phase 3). Confirms the existing design is institutionally recognized — reimplement natively, don't import qlib.contrib.strategy.

3 · Qlib's YAML config-injection triple

{class, module_path, kwargs} as the shape for QuantOS's own strategies_registry/*.yaml (Blueprint Phase 1), if/when that level of runtime flexibility is wanted. Pattern only.

4 · MLflow, adopted directly

Skip Qlib's Recorder wrapper entirely — take MLflow itself when Blueprint Phase 5 (experiment tracking) starts. One well-known dependency, not a framework.

5 · TradingAgents' CI/testing template

Study the pytest-matrix + ruff + clean-install-smoke-test structure when standing up QuantOS's own CI (Blueprint Phase 1). A worked example of "what good CI looks like for a Python trading codebase," nothing to depend on.

6 · Provider-agnostic LLM client pattern (deferred)

If/when the PRD's already-deferred LLM signal layer (Phase 5, post-live-capital) is ever built, mirror TradingAgents' multi-provider client-factory pattern — including Ollama-local support, the only realistic self-hosted option given the RTX 3050's 4GB VRAM ceiling. Not active work today.

07 / 13

Rejected Components

From Qlib — rejected wholesale

  • Binary data store + expression-engine runtime (no NSE support, C-extension Windows build risk)
  • PIT database subsystem (doesn't solve survivorship bias either; would need building the same NSE vendor integration regardless)
  • Backtest engine / Position-Order-Account object graph (QuantOS's own engine already proven adequate)
  • Recorder/MLflow wrapper (take MLflow directly instead)
  • Online serving stack (self-admittedly incomplete, doesn't reach order generation)
  • RL / time-series model variants (tightly coupled to Qlib's internals, not worth the coupling for models not currently in use)

From TradingAgents — rejected wholesale

  • LangGraph debate/consensus architecture (premature complexity at current scale)
  • LLM-as-Portfolio-Manager (zero code-enforced risk math — strictly worse than QuantOS's own designed risk gate)
  • LLM-as-Risk-Manager (three personas debating is not a risk engine)
  • Non-deterministic decision path of any kind, anywhere near the frozen strategy or the live risk gate
  • backtrader dependency and the unreferenced redis dependency (unexplained, unneeded)
Standing rule, not just for this review: no LLM-generated output — from TradingAgents' pattern or any future in-house LLM work — is ever wired as a decision-maker for position sizing, risk limits, or order placement. If an LLM signal layer is ever built (PRD's own deferred Phase 5), its output feeds in as one more analyst-style input to QuantOS's existing deterministic scoring, never as the thing that decides.

Migration Strategy

Sequenced against the existing Target Architecture Blueprint's own Phase 0–9 roadmap — nothing here reorders it. Nothing here touches momentum_backtest.py or paper_trader.py's trading logic.

  1. Immediate — after validation, not before
    "Immediate" means: safe to start once the 13-week Prospective Validation freeze allows non-frozen work, i.e. now, on non-Momentum surfaces only. Port the relevant Alpha158 formula subset into factor_attribution.py's sibling tooling or a new pure-function module, targeting Quality Factor only. Zero risk to the frozen strategy since it's a different, non-frozen codebase path.
  2. Medium term — before paper/live (Blueprint Phase 1–5)
    When Blueprint Phase 1 (config skeleton) is built: shape it around Qlib's DI-config triple. Technical Risks

Overengineering

The single largest real risk of this review existing at all: getting distracted chasing a 46k/93k-star framework instead of finishing the 13-week validation gate that's the project's own stated single milestone. Mitigated only by discipline — the recommendations above are deliberately small and deferred.

Vendor / architecture lock-in

Qlib's binary format + calendar/instrument files, and TradingAgents' LangGraph state-schema coupling, are two different lock-in shapes. Neither is being adopted wholesale, so neither risk materializes — worth stating as the reason wholesale adoption was rejected, not just a caveat.

Maintenance burden / dead-project risk

Qlib: real. Bus-factor-concentrated (~48% one contributor), 3-month-stale push, Microsoft's attention visibly shifting to RD-Agent. A solo operator debugging someone else's C-extension build failure on Windows is exactly the failure mode to avoid. TradingAgents: healthier (bazaar contributors, active CI) but iterates fast — itself a tax on a system that values stability over features.

Dependency explosion

TradingAgents' full stack (langchain + langgraph + multiple provider SDKs + an unreferenced redis dep) is heavy for a 6,000-LOC solo project. Qlib pulls a Cython build chain and version-pinned numpy/scipy that could clash with the existing environment. Neither is being installed, so this risk is avoided by the recommendation itself.

Research reproducibility

The starkest finding in this whole review. TradingAgents is admittedly, permanently non-deterministic on its highest-stakes nodes. Any contamination of this pattern into the strategy layer — even accidentally, even "just for research" — breaks the exact property (§8/§9 of the audit) the project has spent real engineering effort proving.

Breaking deterministic execution / the frozen strategy

Per CONTEXT.md:14-36, any change touching Momentum's alpha model, ranking, universe, or parameters restarts the 13-week validation clock. None of this review's recommendations touch that code — this is the explicit test every recommendation above was run through before being written down.

11 / 13

Final Recommendation

Adopt fragments, reject both frameworks

Neither Qlib nor TradingAgents should be installed as a dependency. Both are legitimate, well-built projects for what they're actually for — a Linux-first institutional research platform, and a multi-agent LLM research demo respectively — and neither of those is what QuantOS, as a solo-operator Windows-laptop NSE momentum system in a frozen validation phase, currently is. Six small, specific ideas are worth harvesting (§6); everything else, including both runtimes wholesale, is rejected (§7) on engineering, licensing-adjacent maintenance, and — for TradingAgents specifically — categorical determinism grounds that no future release of either project changes.

Committee notes

CTO
The right call here isn't "which framework is better" — it's recognizing neither one is solving QuantOS's actual two hard problems. Approve the six harvested fragments, table everything else.

TradingAgents' Portfolio Manager is a worse risk engine than what's already sitting in the unbuilt Blueprint — zero numeric limits, LLM judgment as the final gate. That's a regression, not a feature, if anyone's ever tempted to wire it in.

Alpha158 is worth two to four days of formula-porting for Quality Factor. That's the whole tangible win here — everything else is validation of decisions already made, which has real value but isn't code.
DevOps Architect
Qlib's own docs tell you not to run it on Windows without Docker, and this project has no Docker. That's not a nice-to-have caveat, that's the platform saying no.
OSS Governance
MIT and Apache-2.0 are both fine licenses — this was never a licensing question. It's a fit question, and the answer is the same for both: good project, wrong shape for this system, right now.
