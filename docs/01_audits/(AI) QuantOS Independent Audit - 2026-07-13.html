Execution flow

Manual (or unconfirmed-scheduled) invocation → run_daily_traders.ps1 → Push-Location into each strategy's directory → strategy script fetches live prices via yfinance → computes signal (momentum rank / quality score / regime classification / VRP) → live regime check (uncached, re-fetched every run) → orders queued, not filled → next invocation fills queued orders at that day's price (T+1 design) → trade appended to CSV, state written to JSON → portfolio_tracker.py aggregates a cross-strategy snapshot. No step in this chain touches a broker API.

Development Phase

Phase 7of 14 — Validation Framework

The project's own operating rule, active as of 2026-07-13 (CONTEXT.md:14-36), states plainly where it is: Momentum is frozen pending 13 weekly rebalance observations under a "Prospective Validation" protocol, currently at 0 of 13. That is a Phase 7 (Validation Framework) gate, not a Phase 10 (Paper Trading) achievement — paper trading is the delivery mechanism being used to finish validation, not evidence validation is done.

It would be generous to call this Phase 10. The paper-trading infrastructure works and has run since 2026-06-09 — real logs, real (virtual) P&L, real bug fixes under load. But Phase 8 (Portfolio Construction) is absent (0/24 subsystem score for portfolio optimizer beyond equal-weight-top-10), and Phase 9 (Execution Engine) is completely unbuilt (0% — zero broker code, confirmed by exhaustive grep in §7). A project doesn't get credit for Phase 10 while Phase 8 and 9 haven't started; it has leapfrogged into live-adjacent operation on validation logic alone, which is exactly what a "paper trading" stage is supposed to let you do safely — but the maturity claim belongs to Phase 7, because that's the open question the current activity exists to answer.

Reality check on evidence quality: even within Phase 7, the strongest statistical result (full-sample CAPM alpha, p=0.002) lost significance on the only fair post-2021 window tested (p=0.23) — see §8. The validation phase has not yet produced a clean pass.

Risk Assessment

Critical blockers

  • Universe fabrication (F9). utils/data_loader.py:57-74 — get_nifty500_universe() is a hardcoded 95-ticker list, not an index fetch, including ZOMATO (IPO 2021-11-18), LICI (2022-05-17), DELHIVERY (2022-05-24), PAYTM, IRCTC (2019) — used as the tradeable universe for backtests starting 2010. This is survivorship bias by construction, not by accident, and it feeds 2 of the 4 live tournament strategies (Quality Factor, Factor Timing).
  • Survivorship bias, no point-in-time membership (F1). fetch_universe.py:13 pulls NSE's current constituent list; momentum_backtest.py backtests 2019–2024 against that single, current-day snapshot. Every delisted/merged/dropped name is invisible. Confirmed present in code, not merely a documented risk.
  • SEBI compliance status unknown, deadline already passed. Full enforcement of the Feb 2025 algo-trading circular hit 1 Apr 2026; today is 2026-07-13, over three months past. The compliance checklist in PRD.md:132-154 has 0 of 7 items checked. This blocks any live order placement, not just automation.
  • No portfolio-level risk engine. Four strategies run isolated ₹1L virtual books with no cross-strategy exposure aggregation, correlation check, or kill switch. At go-live, correlated concentrated positions across strategies are invisible to the system by design.

High-priority

  • Cost model inconsistency, sibling repo. Each of the 3 sibling paper traders hardcodes its own flat cost constant (0.003–0.005); none imports the accurate transaction_costs.py model AlgoTrader now uses. Paper P&L for 3 of 4 tournament strategies is optimistic relative to real cost.
  • Zero test coverage on the strategy layer. TASK_FIX_BUGS.md documents 4 confirmed bugs (decimal-threshold, index-vs-value lookup, signal misattribution, options CAGR annualized at 1691%) found only by manual eyeballing. 16 strategy files, 4,513 LOC, have no automated test protecting against recurrence.
  • Confirmed scheduler reliability gaps. trader_logs/: 2026-07-01 has no log at all (a real miss, not a market-holiday skip); 2026-06-30 truncates mid-run with no failure flag; 2026-07-08 triggered over an hour late. The 2026-07-13 fail-loud fix (flag file, exit code, red console banner) addresses detected errors, not silent scheduler non-execution.
  • Blanket exception handling, still open. paper_trader.py (check_regime, fetch_current_prices, compute_momentum_scores) and download_data.py:64 (fully silent, no log at all) use un-narrowed except Exception. The regime-check fallback direction is risk-increasing: any exception defaults to "bull market."

Architectural debt

  • Two independent codebases, zero shared package. A cost-model bug fixed in AlgoTrader (F4) required a second, separate fix pass across three sibling files to reach parity — and only 1 of those 3 files was actually fixed to import a shared, accurate model; the other two still hardcode flat rates.
  • No git repository anywhere in the tree. No commit-SHA-to-result traceability; results/ in the sibling repo has ~100+ CSV/PNG/HTML artifacts with no run manifest tying any of them to the code state that produced them.

Research gaps (see §8 for detail)

  • Momentum's alpha loses significance on the only fair post-2021 window tested (p=0.23) — the earlier significant result depends materially on the 2019–2021 COVID crash/recovery being in-sample.
  • Single-factor confound unresolved: whether the ~30% annualized alpha is genuine security-selection skill or an unmodeled small/mid-cap risk premium has not been tested (the recommended two-factor extension against Nifty Smallcap 250 is proposed, not yet run).

Security risks

Low current exposure (no live credentials exist to leak), but the security posture is untested, not proven: .env.example loading is dead code, and no secrets-handling path has ever been exercised end to end with a real key.

08 / 15

Research Assessment

This is the project's strongest dimension, and it deserves to be described precisely rather than either dismissed or oversold.

What's genuinely rigorous

  • Explicit falsification chain, not cherry-picked confirmation. One-factor CAPM (alpha +30.2% ann., p=0.0021) → two-factor with size control (alpha shrinks 21% to +23.2% ann., p=0.009, survives) → three-factor on the only fair 2021–2024 overlap window (alpha loses significance, p=0.23). The project reports the failed test as the headline finding of its own closure summary rather than burying it.
  • Look-ahead bias found and fixed with proof. Signal/fill separation (T+1) implemented in both engines, with an assert exec_date > date invariant running on every single buy in production, not just in a test file.
  • Determinism proven, not asserted. A Python hash-seed ordering bug (raw set() iteration affecting which stocks get bought when cash is short) was found in all four tournament strategies and fixed; self-checks force different PYTHONHASHSEED values across subprocess runs specifically to exercise the failure mode, not just re-run the same process.
  • Regime-based return decomposition found 54% of total return concentrated in TRENDING_BULL weeks (26% of the sample) — real concentration, reported honestly, with the caveat that TRENDING_BULL recurs every year through 2024 (i.e., "it was all COVID" is explicitly ruled out as the full explanation).

What's unresolved

  • Size-factor confound. The CAPM regression uses raw (not excess-of-risk-free) returns and a single-factor (Nifty 50) benchmark against a Nifty-500-sourced strategy that includes mid/small-caps. The report itself states this "cannot distinguish genuine security-selection skill from small/mid-cap risk premium" — the two-factor extension against a smallcap index is proposed but not yet run.
  • Survivorship bias is accepted, not corrected. Documented as a deliberate decision (5 blocked attempts to source point-in-time NSE constituent data), one-directional (inflationary only) — a defensible call given data-access reality, but it means every absolute CAGR/Sharpe number in the project should be read as an upper bound.
  • Coverage is 1 of 20 strategies. This entire falsification chain (CAPM, factor timing, regime stability, determinism) was applied only to Momentum. The sibling repo's 16 strategies have research quality closer to "backtest + manual bug hunt" (4 confirmed metric-breaking bugs found that way) than to statistical hypothesis testing.
  • Quality Factor's fundamentals gate is a documented no-op. strategies/07_quality_factor.py:211 hardcodes PE to None, making the quality_value PE≤50 filter never fire — confirmed by results/07_quality_metrics.csv showing quality_cagr and quality_value_cagr byte-identical. DEVLOG.md itself flags this as "residual look-ahead bias risk... treat as upper-bound."

Net read: real, non-trivial, honestly-reported statistical signal that has survived two falsification attempts and failed a third on power grounds, not on a clean null result. This is "mixed evidence, competently characterized" — a genuinely unusual place for a solo retail project to land, and better than either "confirmed edge" or "no edge" would be as a summary.

Executive Summary
